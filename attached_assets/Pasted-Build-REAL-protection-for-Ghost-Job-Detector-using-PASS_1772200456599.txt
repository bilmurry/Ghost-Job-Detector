Build REAL protection for Ghost Job Detector using PASSWORDLESS "Secure Link" authentication with minimal user friction. I do NOT want username/password login. I want email-based Secure Link sign-in, and after successful sign-in the user sees a minimal confirmation page that auto-closes so they can return to their job tab.

STACK CONTEXT
- Node/Express TypeScript backend.
- Browser extension calls POST /api/analyze (and potentially other /api/* endpoints).
- The backend is currently public in Replit; I want backend access restricted to authenticated users.
- The extension should remain click-to-analyze (no auto scraping). If not logged in, it must show a “Sign in” CTA and open the login page.

TERMINOLOGY (IMPORTANT)
- User-facing wording MUST be "Secure Link" (never say "magic link" anywhere user-facing).
- UI text: "Request Secure Link", "Resend Secure Link", "Secure Link expires in 2 minutes".

A) PASSWORDLESS SECURE LINK AUTH (NO PASSWORDS)
1) DATABASE
Use the existing db/storage approach already used in this repo (whatever is used for analysis history). Create tables:

- users:
  id (string/uuid), email (unique), createdAt, lastLoginAt

- login_tokens:
  id (string/uuid),
  email,
  tokenHash,
  createdAt,
  expiresAt,
  usedAt (nullable),
  ip (nullable),
  userAgent (nullable)

Token rules:
- Raw token generated by crypto.randomBytes(32).toString("hex")
- Store ONLY tokenHash (sha256 of raw token). Never store raw token.
- Expires in 120 seconds (2 minutes): expiresAt = createdAt + 120s
- Single use: usedAt set on consumption

2) BACKEND ROUTES
Implement these routes:

- POST /api/auth/request-link
  body: { email }
  behavior:
    - Always return 200 { ok: true } (avoid account enumeration)
    - Validate email format server-side
    - Enforce resend cooldown (see section C)
    - If allowed (not in cooldown):
       - create raw token
       - store tokenHash + metadata (email, createdAt, expiresAt, ip, userAgent)
       - send email with Secure Link:
         ${BASE_URL}/api/auth/secure?token=<RAW_TOKEN>
  Note: even when in cooldown, still return 200 { ok: true }.

- GET /api/auth/secure?token=...
  behavior:
    - Hash provided token and lookup login_tokens.tokenHash
    - If not found:
        - render minimal HTML page:
          "Invalid Secure Link. Request a new one."
          button -> /login
    - If usedAt exists:
        - render:
          "This Secure Link has already been used."
          button -> /login
    - If expired:
        - render:
          "This Secure Link has expired."
          button -> /login
    - If valid:
        - set usedAt = now
        - create/find user by email
        - update users.lastLoginAt
        - create authenticated session cookie (server-managed session)
        - redirect to /signed-in

- GET /api/auth/me
  returns JSON:
    - { authenticated: true, email } if logged in
    - { authenticated: false } if not

- POST /api/auth/logout
  clears session and returns { ok: true }

3) PUBLIC PAGES (MINIMAL)
- GET /login
  Provide a simple login page (server-rendered or existing frontend route) with:
    - Email input
    - Button: "Request Secure Link"
  After submission:

C) RESEND SECURE LINK COOLDOWN (BACKEND + FRONTEND)
1) Backend cooldown enforcement (30 seconds)
In POST /api/auth/request-link:
- Query the most recent login_tokens.createdAt for that email
- If last token was created within 30 seconds:
  - DO NOT create a new token
  - DO NOT send email
  - Still return 200 { ok: true }
Optional:
- You may return { ok: true, cooldownSecondsRemaining: N } to drive the countdown, but never reveal whether the email exists.

2) Frontend countdown
On /login in the success state:
- Disable "Resend Secure Link" for 30 seconds with a visible countdown
- After countdown, enable it
